
## HTTP

The primary difference lies in **where and how** the data is transmitted from the browser (client) to the server.

#### 1. The GET Method

* **Data Location:** Data is appended directly to the URL as a query string (e.g., `http://localhost:3000/print?name=John&dob=2000-01-01`).
* **Server Routing:** Because the URL changes dynamically based on user input, the server cannot use a strict string match for routing. Instead, it must look for requests that *start with* the endpoint path (e.g., `startsWith('/print')`).
* **Server Processing:** The server utilizes a URL parsing package (like `url.parse(req.url, true)`) to instantly break the URL down into components and extract the query string as a readable object.

#### 2. The POST Method

* **Data Location:** Data is embedded invisibly inside the **body of the HTTP request message**, completely hidden from the address bar.
* **Server Routing:** The URL remains clean and static (e.g., `http://localhost:3000/enter`), allowing the server to use strict endpoint matching.
* **Server Processing:** Because data in a body can be large, the server treats the request as a readable stream. It must process the incoming payload **one chunk at a time** using an event listener (`request.on('data', ...)`), buffering the chunks into a string, and finally parsing it once the stream emits the `end` event (`request.on('end', ...)`).


### Direct Comparison 

| Feature | GET Method | POST Method |
| --- | --- | --- |
| **Data Visibility** | **Visible:** Appended directly to the browser's address bar/URL. | **Hidden:** Embedded inside the body of the HTTP request message. |
| **Server Routing** | Requires partial matching (e.g., `startsWith()`) because the URL changes with input. | Uses strict matching because the URL path remains identical. |
| **Data Handling** | Extracted instantly by parsing the request URL components. | Streamed sequentially in chunks, needing a buffer and listener events (`data`, `end`). |
| **Use Case Example** | Sending non-sensitive data like search queries, names, or dates. | Submitting sensitive data like passwords or submitting forms that alter data. |

### Critical Security Context 

Your professor highlighted a very common misconception regarding form security that seniors love to test:

* **Why GET is bad for passwords:** You should **never** use GET for passwords or sensitive data because the values are exposed right in the URL address bar for anyone looking at the screen, and they get saved into browser histories or network logs.
* **The POST Security Illusion:** Using `type="password"` hides the characters on the screen, and using `POST` hides the data from the URL bar. **However, POST does not provide encryption**.
* **The Solution:** To actually secure data over the network and protect it from being intercepted, you must use the **HTTPS protocol** alongside the POST method.
